Повторяемый рабочий процесс от контракта до cleanup

ФОКУС: P0 ОСНОВА

Один цикл, который нужно запомнить

КОНТРАКТ
ДАННЫЕ

RESPONSE

DIAGNOSE
  1. Понять method/path/auth/request/response.
  2. Создать контролируемые данные.
  3. Отправить request.
  4. Проверить transport, structure и business data.
  5. После mutation прочитать сохранённое состояние.
  6. Удалить созданное даже после падения.
  7. Оставить данные для поиска причины.

P0Без этого тест не рабочий

  • Конкретный status, а не «любой 2xx».
  • Ключевые значения связаны с request.
  • Собственный id вместо общей записи.
  • Read-back после POST/PUT/PATCH/DELETE.
  • Отказ без auth для защищённой операции.
  • Cleanup и условное логирование.
  • Каждый тест запускается отдельно.

P1Ежедневная работа

DTO/Jackson, JsonPath, specifications, schema, AssertJ для DTO, JUnit parameterization и lifecycle. Они усиливают P0, но не заменяют его.

P2После уверенной базы

Custom filters, сложный GPath, parallel suite, расширенный HTTP config, retries, Allure/CI integrations.

Редкий метод библиотеки не компенсирует отсутствие state check.

Почему status 200 недостаточно

Можно получить 200 с пустым body, чужой записью, неверной ценой или несохранённым PATCH. Status — первый слой, а не финал.

Общий учебный сервис

Restful Booker периодически сбрасывает данные. Нельзя полагаться на постоянный id/размер списка. Создавайте своё и очищайте после теста.

Цель

Взять новый endpoint, самостоятельно выбрать данные и проверки, собрать устойчивый сценарий и объяснить причину падения.

Анатомия request

given()
    .baseUri(BASE_URL)
    .accept(ContentType.JSON)
    .contentType(ContentType.JSON)

3/9

Лестница проверок response

От transport к business data и сохранённому состоянию

ВАЖНОСТЬ: P0

Пять слоёв сверху вниз

1. TransportStatus, content type, важные headers.
2. StructureПоля, JSON types, nesting, schema.
3. BusinessЗначения относятся к request.
4. Saved stateGET подтверждает mutation.
5. Unchanged statePATCH не испортил остальное.

P0Конкретный transport

response.then()
    .statusCode(200)
    .contentType(ContentType.JSON);

200, 201, 204, 400, 403 и 404 — разные результаты. «status < 500» обычно слишком слабо.

Structure + business

response.then()
    .body("bookingid", greaterThan(0))
    .body("booking.firstname",
          equalTo(expected.firstname()))
    .body("booking.bookingdates.checkin",
          equalTo(expected.bookingdates().checkin()));

Проверяйте риск

  • Идентификатор создан.
  • Ответ относится к request.
  • Ключевое поле корректно.
  • Вложенный объект не потерян.
  • Mutation сохранилась.

Response mutation не доказывает state

Недостаточно
PATCH ...
  .then()
  .statusCode(200);
Нужно
PATCH → 200
GET /id → saved
assert saved

Разные mutation — разные ожидания

PUTSaved DTO полностью равен updated.
PATCHChanged field + preserved fields.
DELETEGET после удаления → 404.

Минимум create-теста

  1. POST вернул status и JSON.
  2. bookingid положительный.
  3. Response связан с request.
  4. GET по id вернул booking.
  5. DTO совпал с expected.
  6. id зарегистрирован для cleanup.

Schema — один слой

JSON может соответствовать schema и содержать чужой firstname. Structure не заменяет business.

4/9

Собственные данные: POST → GET → cleanup

Основа независимого теста на общем окружении

ВАЖНОСТЬ: P0

Почему нельзя менять «первый id»

  • Он может принадлежать другому тесту.
  • База может сброситься.
  • Параллельный пользователь изменит/удалит запись.
  • Порядок общего списка нестабилен.

Read-only знакомство может извлечь текущий id. Mutation — только на своём.

Создать и извлечь id

BookingRequest expected = uniqueBooking("Create");

int bookingId = given()
    .spec(BASE_SPEC)
    .contentType(ContentType.JSON)
    .body(expected)
  .when()
    .post("/booking")
  .then()
    .statusCode(200)
    .extract().path("bookingid");

createdBookingIds.add(bookingId);

Прочитать saved DTO

BookingResponse actual = given()
    .spec(BASE_SPEC)
    .pathParam("id", bookingId)
  .when()
    .get("/booking/{id}")
  .then()
    .statusCode(200)
    .extract().as(BookingResponse.class);

assertThat(actual)
    .usingRecursiveComparison()
    .isEqualTo(expected);

Уникальные данные

String firstName = "ApiTest-" + UUID.randomUUID();
  • Префикс показывает владельца.
  • UUID снижает конфликт.
  • Значение нужно сохранить в отчёт.
  • Остальные поля валидны и понятны.

Cleanup через JUnit

@AfterEach
void cleanup() {
    for (int id : createdBookingIds) {
        deleteBookingIfPresent(id);
    }
    createdBookingIds.clear();
}

Регистрируйте id сразу после создания, а не в конце сценария.

Страховка try/finally

Integer id = null;
try {
    id = createBooking();
    // GET → PATCH → GET → DELETE → GET 404
} finally {
    if (id != null) deleteBookingIfPresent(id);
}

Cleanup терпим к уже удалённому resource и не скрывает main failure.

Инварианты независимости

  • Создаёт всё, что изменяет.
  • Удаляет только своё.
  • Не зависит от порядка.
  • Параллельные id не смешиваются.
  • Запускается отдельно.

Как выбрать способ проверки body

Hamcrest, extraction, JsonPath, DTO, AssertJ и Schema

ФОКУС: P1 ПОСЛЕ P0

Дерево решения

1 значение дальшеextract().path
2–4 простых поляHamcrest/JsonPath
Объект целикомDTO + AssertJ
Форма/typesSchema дополнительно
Список/filterJsonPath или DTO list

Inline Hamcrest

response.then()
    .statusCode(200)
    .body("firstname", equalTo(expected.firstname()))
    .body("totalprice", equalTo(expected.totalprice()))
    .body("bookingdates.checkin",
          equalTo(expected.bookingdates().checkin()));

Компактно для малого контракта. Десять полей превращаются в стену JsonPath.

Extraction и JsonPath

int id = response.then()
    .statusCode(200)
    .extract().path("bookingid");

String first = response.jsonPath().getString("firstname");
int price = response.jsonPath().getInt("totalprice");

Extraction связывает шаги flow; JsonPath достаёт точечные значения.

Антипаттерн

Извлечь десять полей вручную и собрать объект. Если response — связанный объект, десериализуйте сразу.

DTO + AssertJ

BookingResponse actual =
    response.as(BookingResponse.class);

assertThat(actual)
    .usingRecursiveComparison()
    .isEqualTo(expected);

Основной вариант для полного booking: типы, IDE и понятные различия.

SoftAssertions для PATCH

SoftAssertions.assertSoftly(softly -> {
  softly.assertThat(saved.lastname())
      .as("changed").isEqualTo(changed);
  softly.assertThat(saved.firstname())
      .as("preserved name").isEqualTo(original.firstname());
  softly.assertThat(saved.bookingdates())
      .as("preserved dates").isEqualTo(original.bookingdates());
});

Schema + business

response.then()
    .body(matchesJsonSchemaInClasspath(
        "schemas/lesson01/booking-schema.json"))
    .body("firstname", equalTo(expected.firstname()));
ЛовитRequired, type, nesting, format.
Не ловитЧужие values, state, auth.

Правило читаемости

Способ проверки должен делать контракт очевиднее. Иначе извлеките DTO или упростите.

Specifications, архитектура и диагностика

Убираем шум и сохраняем бизнес-смысл

ФОКУС: P1

RequestSpecification

RequestSpecification baseSpec =
  new RequestSpecBuilder()
    .setBaseUri(BASE_URL)
    .setAccept(ContentType.JSON)
    .addFilter(new ErrorLoggingFilter())
    .build();

Уместно

  • base URI и safe headers;
  • Accept/Content-Type группы;
  • timeouts/config;
  • failure logging;
  • technical auth клиента.

Не прячьте в spec

  • Данные конкретного теста.
  • Business assertions.
  • Status для разных исходов endpoint.
  • Изменяемый глобальный body.
  • Секреты в открытом логировании.

ResponseSpecification

ResponseSpecification okJson =
  new ResponseSpecBuilder()
    .expectStatusCode(200)
    .expectContentType(ContentType.JSON)
    .build();

response.then()
    .spec(okJson)
    .body("firstname", equalTo(expected.firstname()));

Разделение слоёв

TestsСценарий, expected, assertions.
ClientВыполнение endpoint без business asserts.
ModelsRequest/response DTO.
FixturesВалидные уникальные данные.
Specs/configURL, headers, HTTP settings, logging.

Логировать при провале

given()
    .log().ifValidationFails()
  .when()
    .get("/booking/{id}", id)
  .then()
    .log().ifValidationFails();

Глобальный log().all() создаёт шум и может раскрыть token/password.

Минимум диагностики

  1. Environment/base URL.
  2. Method, path, params.
  3. Safe request headers/body.
  4. Response status/headers/body.
  5. Созданный id и unique data.
  6. Expected/actual и шаг flow.

Auth и изменение состояния

Token, PUT, PATCH, DELETE и обязательный read-back

ВАЖНОСТЬ: P0 + P1

Получить и использовать token

String token = given()
    .spec(BASE_SPEC)
    .contentType(ContentType.JSON)
    .body(new AuthRequest("admin", "password123"))
  .when()
    .post("/auth")
  .then()
    .statusCode(200)
    .body("token", notNullValue())
    .extract().path("token");

given().cookie("token", token);

P0Negative auth

  • Неверный password → Bad credentials.
  • Неверный username → Bad credentials.
  • PUT/PATCH/DELETE без token → 403.

Happy path не доказывает защиту endpoint.

PUT: полная замена

  1. Создать original.
  2. Собрать полный updated DTO.
  3. PUT с token.
  4. Проверить response.
  5. GET saved.
  6. Recursive compare с updated.

Секреты

Token, Authorization и password маскируются в CI/Allure. Failure logging безопаснее глобального.

PATCH: changed + preserved

Map<String, Object> patch =
    Map.of("lastname", changedLastName);

given()
    .spec(BASE_SPEC)
    .cookie("token", token)
    .contentType(ContentType.JSON)
    .body(patch)
  .when()
    .patch("/booking/{id}", bookingId)
  .then()
    .statusCode(200);

BookingResponse saved =
    getBooking(bookingId).as(BookingResponse.class);

DELETE: response + state

given()
    .spec(BASE_SPEC)
    .cookie("token", token)
    .pathParam("id", bookingId)
  .when()
    .delete("/booking/{id}")
  .then()
    .statusCode(201);

getBooking(bookingId).then().statusCode(404);

Что забывают

  • PUT отправляет неполный body.
  • PATCH не подтверждается GET.
  • Не проверены preserved fields.
  • DELETE не подтверждён 404.
  • Изменяется чужая запись.
  • Token печатается в лог.

Формула

before state → authorized action → response → after state → invariants.

JUnit и AssertJ внутри REST suite

Выполнение, test data и понятные проверки одного сценария

ФОКУС: P1

JUnit формирует suite

@TestОдин independent scenario.
@ParameterizedTestОдин contract, таблица inputs.
@BeforeEachFixture/config.
@AfterEachCleanup resources.
@Tagsmoke/regression/api.

Invalid auth как data table

@ParameterizedTest
@CsvSource({
  "admin, wrong-password",
  "wrong-user, password123",
  "wrong-user, wrong-password"
})
void invalidAuth(String username, String password) {
    Response r = auth(username, password);
    assertThat(r.statusCode()).isEqualTo(200);
    assertThat(r.jsonPath().getString("reason"))
        .isEqualTo("Bad credentials");
}

Не связывайте методы

Плохо

testUpdate использует static id из testCreate.

Хорошо

Каждый тест создаёт своё; CRUD живёт внутри одного метода.

Параллельность начинается с дизайна

Unique data, локальный expected, нет static mutable id, cleanup только своего resource.

AssertJ проверяет Java DTO

BookingResponse actual =
    response.as(BookingResponse.class);

assertThat(actual)
    .usingRecursiveComparison()
    .isEqualTo(expected);

REST Assured проверил HTTP, Jackson создал DTO, AssertJ показал различия.

Soft assertions для связанных полей

SoftAssertions.assertSoftly(softly -> {
  softly.assertThat(saved.lastname())
      .as("changed").isEqualTo(changed);
  softly.assertThat(saved.firstname())
      .as("preserved").isEqualTo(original.firstname());
  softly.assertThat(saved.totalprice())
      .as("price").isEqualTo(original.totalprice());
});

Что где проверять

REST AssuredStatus/type/headers/small body.
AssertJDTO, lists, recursive/soft.
JUnitExecution/lifecycle/data/report.

Не делайте assert true

Проверяйте actual-данные типовыми assertions. Иначе теряется причина расхождения.

9/9

Финальный boss и алгоритм работы

Самостоятельный CRUD-flow и критерий готовности

ИТОГ: P0 СОБРАН

Финальный independent flow

  1. Создать unique request DTO.
  2. POST, transport/body, extract id.
  3. Сразу зарегистрировать cleanup.
  4. GET, DTO, recursive compare.
  5. PATCH одного поля с token.
  6. GET, changed + preserved.
  7. DELETE с token.
  8. GET → 404.
  9. finally страхует cleanup.

Скелет

BookingRequest initial = uniqueBooking("Final");
Integer id = null;
try {
    id = create(initial);             // POST
    assertBooking(id, initial);       // GET + AssertJ

    patchNeeds(id, "Lunch", token);   // PATCH
    assertNeeds(id, "Lunch");         // GET

    delete(id, token);                // DELETE
    get(id).then().statusCode(404);   // state
} finally {
    if (id != null) deleteIfPresent(id);
}

Локализуйте шаг

POSTPayload/type/create contract.
GET 1Persistence/DTO/data.
PATCHAuth/partial update.
GET 2Changed/preserved state.
DELETE/404Deletion/state.
CleanupInfra; не скрывать main error.

Алгоритм нового endpoint

  1. Выписать method/path/auth/request/response.
  2. Определить риск и state transition.
  3. Спроектировать independent data.
  4. Вручную изучить real response.
  5. Автоматизировать happy path.
  6. Для mutation добавить read-back.
  7. Добавить ценные negative cases.
  8. Добавить cleanup/failure logs.
  9. Только затем вынести reuse.
  10. Запустить отдельно и в suite.

Checklist review

  • □ Contract, не только доступность.
  • □ Expected связан с request.
  • □ Нет чужого/static id.
  • □ Mutation подтверждена GET.
  • □ PATCH проверяет preserved.
  • □ Auth positive + negative.
  • □ Cleanup переживает failure.
  • □ Логи без секретов.
  • □ Specs не скрывают business.
  • □ Нет зависимости от порядка.

«Спокойно использую в работе»

Не нужно помнить каждый метод. Нужно без подсказки спроектировать данные, request, assertions, state verification, negative coverage, cleanup и диагностику.

Главный приоритет

Простой independent тест с read-back лучше сложного fluent-кода, проверяющего только status.

Дальше P2

OAuth, custom filters, parallel suite, service virtualization, CI secrets, Allure attachments, HTTP tuning — по потребности проекта.
