3/6 · Конфигурация и диагностика

РАБОТА · tasks06–07

Request specification

new RequestSpecBuilder()
 .setBaseUri(BASE_URL)
 .setAccept(ContentType.JSON)
 .addFilter(new ErrorLoggingFilter())
 .build();

URL, safe headers, config и failure logging — в spec. Body и expected — в тесте.

Negative response

Response r = given().spec(BASE_SPEC)
 .pathParam("id", unknownId)
 .log().ifValidationFails()
 .get("/booking/{id}");
r.then().log().ifValidationFails();
assertThat(r.statusCode()).isEqualTo(404);

Диагностика

  1. Method, URL и params.
  2. Безопасные headers/body.
  3. Status, content type, response body.
  4. Свой ID и expected DTO.
  5. Какой шаг flow упал.

6/6 · Устойчивый CRUD-flow

MIDDLE+ · tasks14–15

Финальный flow

  1. Unique request DTO.
  2. POST → response DTO + ID.
  3. Зарегистрировать cleanup.
  4. GET → recursive comparison.
  5. PATCH → GET changed/preserved.
  6. DELETE → GET 404.

Независимость

  • Нет static mutable ID.
  • Каждый тест создаёт своё.
  • Expected локален.
  • Cleanup работает после падения.
  • Тест проходит отдельно и в suite.

Архитектура после повторений

testsscenario + assertions
clientendpoint calls, возвращает Response
modelsrequest/response DTO
fixturesunique valid data
specsобщая HTTP config

Критерий готовности

Самостоятельно построить request, DTO, assertions, read-back, negative state check, cleanup и понять причину падения.
