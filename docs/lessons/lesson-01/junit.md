# JUnit 5: практическая основа для автотестера

JUnit — это каркас тестов. Он не отправляет HTTP-запросы и не сравнивает сложные объекты
так удобно, как AssertJ. Его основная работа — найти тесты, подготовить окружение,
запустить методы нужное количество раз и выполнить очистку после завершения.

Задания находятся в `JUnitTasksTest.java`. В них используется небольшая имитация API,
чтобы изучение JUnit не зависело от интернета и состояния публичного сервера. Смысл
сценариев остаётся рабочим: авторизация, создание данных, чтение и очистка.

В основной блок намеренно не включены `@MethodSource`, собственные расширения JUnit,
тайм-ауты и условный пропуск тестов. Эти возможности существуют, но для начала работы
они нужны гораздо реже, чем обычные тесты, подготовка данных и параметризация.

## 1. Что делает JUnit

Когда вы нажимаете зелёную кнопку около теста, происходит примерно следующее:

```text
JUnit находит метод с @Test
        ↓
запускает подготовку @BeforeEach
        ↓
выполняет тестовый метод
        ↓
выполняет очистку @AfterEach
        ↓
показывает результат в IntelliJ и отчёте
```

Если проверка не прошла, JUnit помечает тест красным. Если все проверки прошли — зелёным.
Если на методе стоит `@Disabled`, тест будет найден, но пропущен.

## 2. Обычный тест с `@Test`

Минимальный тест выглядит так:

```java
@Test
void activeUserCanLogIn() {
    LoginResult result = authService.login("ivan", "secret");

    assertEquals(200, result.statusCode());
}
```

`@Test` сообщает JUnit, что метод нужно запускать как тест. Без этой аннотации обычный
метод внутри тестового класса автоматически не запустится.

Хорошее имя метода описывает поведение:

```text
activeUserCanLogIn
unknownUserReceivesUnauthorizedStatus
createdOrderCanBeReadById
```

Имя `test1` не объясняет, что сломалось. Номер допустим в учебном префиксе `task01`, но
после него всё равно должно быть понятное описание сценария.

## 3. Три части теста: подготовка, действие, проверка

Тест удобно мысленно разделять на три части.

```java
@Test
void productIsCreated() {
    // Подготовка: создаём ожидаемые данные
    ProductRequest request = new ProductRequest("Keyboard", 5000);

    // Действие: вызываем проверяемую систему
    ProductResponse response = productService.create(request);

    // Проверка: сравниваем ожидаемое с фактическим
    assertEquals("Keyboard", response.name());
    assertEquals(5000, response.price());
}
```

В англоязычной литературе это называют Arrange — Act — Assert. Термины знать полезно,
но важнее понимать смысл:

- сначала известны входные данные и ожидания;
- затем выполняется одно главное действие;
- после этого проверяется наблюдаемый результат.

Не смешивайте подготовку и проверки в длинную цепочку. При падении теста должно быть
понятно, не создались данные, не выполнилось действие или не совпал результат.

## 4. Основные проверки JUnit

Чаще всего встречаются следующие методы:

```java
assertEquals(expected, actual);
assertTrue(condition);
assertFalse(condition);
assertNull(value);
assertNotNull(value);
```

### В каком порядке передавать значения

В `assertEquals` сначала идёт ожидаемое значение, затем фактическое:

```java
assertEquals(200, response.statusCode());
```

Так сообщение об ошибке читается естественно: ожидалось `200`, получено `500`.

### Когда использовать `assertTrue`

`assertTrue` подходит для настоящего логического условия:

```java
assertTrue(createdId > 0);
```

Но для обычного равенства лучше `assertEquals`:

```java
assertEquals("Ivan", response.name());
```

Так отчёт об ошибке будет информативнее, чем у проверки вида
`assertTrue("Ivan".equals(response.name()))`.

## 5. Несколько проверок одного ответа: `assertAll`

Ответ API часто нужно проверить по нескольким связанным признакам: статус, ID и данные.
Для этого можно использовать `assertAll`:

```java
assertAll("Создание пользователя",
        () -> assertEquals(201, response.statusCode()),
        () -> assertTrue(response.body().id() > 0),
        () -> assertEquals("Ivan", response.body().name()));
```

Без `assertAll` выполнение остановится на первой упавшей проверке. С `assertAll` JUnit
выполнит все вложенные проверки и покажет их ошибки вместе.

Переменная внутри `() -> ...` не содержит скрытого объекта. Это короткая функция без
аргументов, которую JUnit вызовет позже. Благодаря этому JUnit может собрать несколько
ошибок вместо остановки на первой.

Используйте `assertAll`, когда проверки описывают один результат. Не объединяйте в него
несколько несвязанных сценариев только ради одного зелёного теста.

## 6. Подготовка перед каждым тестом: `@BeforeEach`

API-тестам часто нужны одинаковые начальные объекты:

- клиент API;
- данные пользователя;
- DTO запроса;
- уникальное имя;
- настройки окружения.

Повторяющуюся подготовку можно вынести в метод с `@BeforeEach`:

```java
private UserRequest request;

@BeforeEach
void prepareUser() {
    request = new UserRequest(
            "User-" + UUID.randomUUID(),
            true);
}
```

JUnit вызовет `prepareUser()` перед каждым включённым тестом класса. Каждый тест получит
новый объект, поэтому изменение данных в одном сценарии не перейдёт в другой.

### Что не следует помещать в `@BeforeEach`

Не прячьте там главное действие теста. Если сценарий проверяет создание заказа, сам вызов
`createOrder()` должен быть виден в тесте. В подготовку выносите только общие исходные
данные и техническую настройку.

## 7. Очистка после каждого теста: `@AfterEach`

Автотест не должен оставлять после себя пользователей, заказы или бронирования. Обычно
тест запоминает созданные ID, а `@AfterEach` удаляет их:

```java
private final List<Integer> idsForCleanup = new ArrayList<>();

@AfterEach
void cleanup() {
    for (int id : idsForCleanup) {
        api.deleteUser(id);
    }
    idsForCleanup.clear();
}
```

После создания запись регистрируется для очистки:

```java
int userId = response.body().id();
idsForCleanup.add(userId);
```

Главное преимущество `@AfterEach`: JUnit вызовет очистку даже после падения проверки.
Если написать удаление только в конце теста, оно не выполнится, когда более ранний
`assertEquals` выбросит ошибку.

### Практическое правило

```text
создал данные → сразу сохрани ID для очистки
```

Не откладывайте регистрацию ID до конца теста.

## 8. Параметризованный тест

Иногда один и тот же сценарий нужно выполнить с разными входными данными. Вместо трёх
почти одинаковых тестов можно написать один `@ParameterizedTest`.

JUnit будет запускать метод несколько раз. Каждый запуск является отдельным результатом
в отчёте.

Параметризация подходит, когда меняются только данные, а шаги и смысл проверки одинаковы.
Если сценарии требуют разных запросов или разных проверок, лучше оставить отдельные тесты.

## 9. Один параметр: `@ValueSource`

Для набора простых значений используется `@ValueSource`:

```java
@ParameterizedTest
@ValueSource(ints = {0, -1, -100})
void nonPositiveIdIsRejected(int id) {
    ApiResponse response = api.getUser(id);

    assertEquals(400, response.statusCode());
}
```

JUnit выполнит этот метод три раза:

```text
id = 0
id = -1
id = -100
```

Тип параметра метода должен подходить к данным источника: `ints` передаются в `int`,
`strings` — в `String`.

## 10. Несколько параметров: `@CsvSource`

Если один сценарий требует нескольких входов и ожидаемого результата, подходит
`@CsvSource`:

```java
@ParameterizedTest
@CsvSource({
        "user, correct, 200",
        "user, wrong, 401",
        "unknown, correct, 401"
})
void authenticationCases(String login, String password, int expectedStatus) {
    ApiResponse response = api.login(login, password);

    assertEquals(expectedStatus, response.statusCode());
}
```

Каждая строка — один запуск. Значения идут в параметры метода слева направо:

```text
первое значение  → login
второе значение  → password
третье значение  → expectedStatus
```

`CsvSource` здесь не связан с настоящим CSV-файлом. Это просто компактная таблица
тестовых данных внутри аннотации.

## 11. `null`, пустая строка и пробел

Обязательные текстовые поля часто нужно проверить минимум тремя значениями:

- `null` — значения нет вообще;
- `""` — пустая строка;
- `" "` — строка содержит только пробел.

JUnit позволяет объединить источники:

```java
@ParameterizedTest
@NullSource
@EmptySource
@ValueSource(strings = {" "})
void nameIsRequired(String name) {
    // один и тот же сценарий для трёх граничных значений
}
```

Это практичнее, чем сложный источник данных, если тесту нужен всего один строковый
параметр.

## 12. Читаемое имя: `@DisplayName`

Имя Java-метода не всегда удобно читать в отчёте. `@DisplayName` позволяет добавить
человеческое описание:

```java
@Test
@DisplayName("Сервис авторизации доступен")
void authServiceIsAvailable() {
}
```

Java-имя всё равно должно оставаться понятным. `@DisplayName` дополняет его, а не
исправляет плохое имя.

## 13. Группы тестов: `@Tag`

Тег позволяет отнести тест к набору:

```java
@Test
@Tag("smoke")
void serviceIsAvailable() {
}
```

Частые теги:

- `smoke` — быстрые критичные проверки;
- `regression` — полный регрессионный набор;
- `api` — тесты API;
- `ui` — тесты интерфейса.

Тег сам ничего не запускает и не пропускает. Он только ставит метку, которую затем
использует Maven, IntelliJ или система непрерывной интеграции.

## 14. Зачем в учебных тестах стоит `@Disabled`

Нерешённые задания содержат `null` и нулевые значения. Если запустить весь проект, они
закономерно упадут. Поэтому каждый учебный тест временно отключён:

```java
@Test
@Disabled
void unfinishedTask() {
}
```

Рабочий порядок:

1. удалить `@Disabled` только у текущего задания;
2. решить его;
3. запустить один тест;
4. после решения нескольких задач запустить весь класс.

В настоящем проекте отключённый тест должен иметь понятную причину и задачу на
восстановление. Бесконечно хранить красный сценарий под `@Disabled` — плохая практика.

## 15. Почему в заданиях нет `@MethodSource`

`@MethodSource` нужен, когда в параметризованный тест передаются сложные Java-объекты.
Он полезен, но не является обязательным для ежедневного старта. Если простые значения
помещаются в `@ValueSource` или `@CsvSource`, отдельный метод с `Stream<Arguments>` только
усложняет чтение.

Принцип выбора источника:

```text
одно простое значение        → ValueSource
несколько простых столбцов   → CsvSource
null и пустая строка         → NullSource + EmptySource
сложные объекты              → MethodSource, когда он действительно понадобится
```

Мы вернёмся к `MethodSource` в отдельном уроке, когда появится реальная задача передавать
несколько полноценных DTO, а не использовать возможность ради самой возможности.

## 16. Разбор заданий

### Задание 1. Обычный тест и проверки ответа

`response` уже получен. Нужно прочитать его компоненты через методы `statusCode()` и
`body()`. Токен находится внутри тела, поэтому путь к нему состоит из двух обращений:
сначала получить тело, затем поле токена.

Здесь `assertAll` объединяет проверки одного ответа авторизации.

### Задание 2. Подготовка через `@BeforeEach`

Аннотация ставится над `prepareBookingForTest`. Внутри метода нужно присвоить полю
`standardBooking` новый `BookingRequest`. Для уникального имени используйте уже знакомый
`UUID.randomUUID()`.

Тест специально не создаёт DTO самостоятельно: он доказывает, что подготовка была
выполнена JUnit до входа в тестовый метод.

### Задание 3. Очистка через `@AfterEach`

Аннотация ставится над `removeCreatedBookings`. Логика похожа на обычную уборку:

```text
взять каждый сохранённый ID → удалить запись → очистить список ID
```

Тест проверяет, что запись действительно существует во время сценария. После выхода из
метода очистку должен запустить JUnit.

### Задание 4. `ValueSource`

Запрос уже выполняется для текущего `bookingId`. Нужно извлечь фактический статус.
Обратите внимание в окне запуска: один метод появится несколькими отдельными запусками.

### Задание 5. `CsvSource`

Сначала вызовите `API.authenticate(username, password)`. Статус берётся напрямую из
ответа. Наличие токена нужно определить безопасно: при ошибке тело ответа может быть
`null`, поэтому нельзя сразу вызывать `response.body().token()`.

### Задание 6. Источники пустых значений

DTO запроса уже создан с текущим `firstname`. Отправьте его через `createBooking`, затем
получите статус. Все три запуска должны подтвердить одинаковое правило обязательного поля.

### Задание 7. Имя и тег

Добавьте над методом две аннотации: одну для текста в отчёте, вторую для группы `smoke`.
После этого вызовите `healthCheck()` и получите статус.

### Задание 8. Итоговый сценарий

Сначала создайте запись. Сразу сохраните ID в `bookingIdsForCleanup`. Затем выполните
`getBooking(id)` и заполните переменные фактическими значениями. Все проверки относятся
к одному сценарию «создали и смогли прочитать».

## 17. Частые ошибки

### Метод подготовки не запускается

Проверьте, что над ним стоит `@BeforeEach`, а не только комментарий или импорт класса.

### Очистка написана, но данные остаются

Проверьте три места:

1. над методом стоит `@AfterEach`;
2. ID добавляется в список сразу после создания;
3. метод проходит по всем ID и очищает список.

### Параметризованный тест не запускается

У метода должны быть и `@ParameterizedTest`, и хотя бы один источник данных.

### В `CsvSource` параметры перепутались

Сопоставьте значения строки и параметры метода слева направо. Количество значений должно
совпадать с количеством параметров.

### Появился `NullPointerException` при проверке ошибки авторизации

При неуспешном ответе `body()` может вернуть `null`. Сначала проверьте существование тела,
и только затем читайте токен.

### Тесты проходят по одному и мешают друг другу вместе

Обычно это означает общие изменяемые данные или отсутствующую очистку. Каждый тест должен
создавать собственные данные и не рассчитывать на порядок запуска.

## Критерий готовности

Практический блок JUnit пройден, если вы можете:

1. написать обычный тест с понятным именем;
2. разделить подготовку, действие и проверки;
3. выбрать подходящий `assert` и не путать expected с actual;
4. подготовить свежие данные через `@BeforeEach`;
5. гарантировать очистку через `@AfterEach`;
6. использовать `@ValueSource` для одного параметра;
7. использовать `@CsvSource` для таблицы входов и ожиданий;
8. отдельно проверить `null`, пустую строку и пробел;
9. добавить понятное имя и тег для отчёта и набора запуска;
10. объяснить, зачем JUnit нужен в тесте, где HTTP отправляет REST Assured,
    а сложные проверки выполняет AssertJ.

`docs/print/junit-cheatsheet/junit-cheatsheet.pdf`.
