# JUnit 5: понятная основа для работы автотестера

Этот материал связан с заданиями из `JUnitTasksTest.java`. Сначала прочитайте
нужный раздел, затем решайте задание с тем же номером.

Главный принцип этого блока: если задание посвящено JUnit, вызов API и получение
фактических данных уже написаны. Вы настраиваете именно JUnit: проверки, жизненный
цикл, количество запусков, источники параметров, имена и теги тестов.

## 1. За что отвечает JUnit

В API-автотесте участвуют несколько инструментов, и у каждого своя работа:

```text
REST Assured отправляет HTTP-запрос и получает ответ
                         ↓
Jackson превращает JSON в Java-объект
                         ↓
JUnit находит и запускает тестовый метод
                         ↓
JUnit Assertions или AssertJ сравнивают ожидаемое с фактическим
```

JUnit — это каркас тестов. Он умеет:

- находить методы, которые являются тестами;
- выполнять подготовку перед тестом;
- запускать один метод несколько раз с разными параметрами;
- выполнять очистку после теста;
- собирать результат каждого запуска;
- добавлять тестам понятные имена и теги.

Сам по себе JUnit не отправляет HTTP-запрос. Поэтому строка вида
`API.authenticate(...)` в учебном классе изображает действие тестируемой системы,
а аннотации вокруг метода показывают работу JUnit.

## 2. Как устроен обычный тест

```java
@Test
void activeUserCanLogIn() {
    LoginResponse response = api.login("ivan", "secret");

    assertEquals(200, response.statusCode());
}
```

`@Test` сообщает JUnit: этот метод нужно запускать как тест. Без `@Test` это
обычный Java-метод, и JUnit сам его не вызовет.

Внутри теста обычно есть три смысловые части:

```text
Подготовка данных → Действие → Проверки
```

Например:

```java
@Test
void orderCanBeCreated() {
    OrderRequest request = new OrderRequest("Book", 2); // подготовка

    OrderResponse response = api.createOrder(request);  // действие

    assertEquals(201, response.statusCode());            // проверка
}
```

Комментарии `given/when/then` писать необязательно. Важно, чтобы границы этих
частей были видны по структуре кода.

## 3. Проверки JUnit и `assertAll`

Самые нужные проверки JUnit:

```java
assertEquals(expected, actual);
assertNotNull(actual);
assertTrue(condition);
assertFalse(condition);
```

Сначала передаётся ожидаемое значение, потом фактическое:

```java
assertEquals(200, response.statusCode());
```

Если у одного ответа проверяется несколько важных свойств, удобно объединить их
через `assertAll`:

```java
assertAll("Создание пользователя",
        () -> assertEquals(201, statusCode),
        () -> assertNotNull(body),
        () -> assertTrue(userId > 0));
```

Что здесь происходит:

1. `assertAll` получает название группы проверок.
2. Каждая `() -> ...` хранит одну проверку, которую JUnit сможет вызвать.
3. JUnit выполняет все переданные проверки.
4. Если упали несколько проверок, в результате будут видны все их ошибки.

Без `assertAll` выполнение метода остановилось бы на первом упавшем `assert`.

Не нужно помещать в `assertAll` разные бизнес-сценарии. Он подходит для нескольких
проверок одного результата: например, статуса, тела и идентификатора одного ответа.

### Как решить задание 1

В задании ответ уже получен, а статус, тело и токен уже разложены по переменным.
Нужно только добавить внутрь подготовленного `assertAll` три лямбды с проверками.

Собирайте решение из этих отдельных элементов:

```java
() -> assertEquals(ожидаемое, фактическое)
() -> assertNotNull(объект)
() -> assertFalse(строка.isBlank())
```

Подставьте переменные из задания и расставьте запятые между лямбдами. Сам вызов
API переписывать не требуется.

## 4. Подготовка перед каждым тестом: `@BeforeEach`

Часто нескольким тестам нужны одинаковые начальные данные. JUnit может создавать
их автоматически перед каждым тестовым методом:

```java
private UserRequest user;

@BeforeEach
void prepareUser() {
    user = new UserRequest("user-" + UUID.randomUUID());
}

@Test
void userCanBeCreated() {
    assertNotNull(user);
}
```

Порядок выполнения:

```text
@BeforeEach → первый тест → @AfterEach
@BeforeEach → второй тест → @AfterEach
@BeforeEach → третий тест → @AfterEach
```

Ключевое слово здесь — `Each`: метод подготовки выполняется заново для каждого
теста. Это уменьшает зависимость тестов друг от друга.

В `@BeforeEach` обычно помещают:

- создание свежего DTO запроса;
- подготовку уникальных тестовых значений;
- получение данных, которые нужны почти каждому тесту класса.

Не стоит прятать туда половину бизнес-сценария. Если конкретный запрос нужен
только одному тесту, его легче понять, когда он остаётся в этом тесте.

### Как решить задание 2

Метод `prepareBookingForTest` уже создаёт новый `BookingRequest`. Нужно выбрать
аннотацию жизненного цикла, которая запускает метод перед каждым тестом, и поставить
её над методом. После этого `task02` увидит непустой `standardBooking`.

То есть ученик не пишет создание DTO: учебная цель — подключить готовую подготовку
к жизненному циклу JUnit.

## 5. Очистка после каждого теста: `@AfterEach`

API-тесты часто создают пользователей, заказы и бронирования. Если их не удалять,
тестовый стенд загрязняется, а повторные запуски начинают мешать друг другу.

```java
private final List<Integer> createdIds = new ArrayList<>();

@AfterEach
void deleteCreatedUsers() {
    createdIds.forEach(api::deleteUser);
    createdIds.clear();
}
```

Когда тест создаёт запись, её идентификатор сразу сохраняют:

```java
int id = api.createUser(request).id();
createdIds.add(id);
```

JUnit запускает `@AfterEach` после тестового метода, в том числе если одна из
проверок теста упала. Поэтому очистка надёжнее, чем вызов удаления в последней
строке самого теста: до последней строки выполнение может не дойти.

### Как решить задание 3

Логика удаления уже написана в `removeCreatedBookings`. Нужно выбрать аннотацию
жизненного цикла для автоматического запуска после каждого теста. В `task03` ID
уже добавляется в список очистки. Менять тело теста или писать цикл не нужно.

## 6. Что такое параметризованный тест

Допустим, одно правило нужно проверить для трёх значений:

```text
id = 0   → статус 400
id = -1  → статус 400
id = -50 → статус 400
```

Можно написать три почти одинаковых метода. Но удобнее дать JUnit один метод и
три набора данных. Тогда JUnit сам выполнит три отдельных запуска.

Для этого нужны две вещи:

1. `@ParameterizedTest` — сообщает, что метод нужно запускать с параметрами.
2. Источник данных — сообщает, какие значения передавать.

Схема работы:

```text
источник отдаёт первое значение  → JUnit вызывает test(первое значение)
источник отдаёт второе значение  → JUnit вызывает test(второе значение)
источник отдаёт третье значение  → JUnit вызывает test(третье значение)
```

Важно: параметр метода не появляется из воздуха. Его значение передаёт JUnit из
указанного источника. Каждый набор будет отдельным тестом в IntelliJ и отчёте.

### Имя отдельного запуска

У `@ParameterizedTest` есть необязательный параметр `name`:

```java
@ParameterizedTest(name = "Проверяем идентификатор {0}")
```

Это не заголовок метода и не данные. Это шаблон текста, который IntelliJ и отчёт
покажут для каждого запуска.

- `{0}` — первый параметр метода;
- `{1}` — второй параметр;
- `{2}` — третий параметр;
- `{index}` — порядковый номер запуска.

Например, при значениях `0` и `-1` будут видны два запуска:

```text
Проверяем идентификатор 0
Проверяем идентификатор -1
```

В заданиях 4 и 5 шаблон имени специально не написан заранее. Его добавляет ученик,
потому что настройка понятного отчёта — часть работы с JUnit.

## 7. `@ValueSource`: одно простое значение на запуск

`@ValueSource` подходит, когда у метода один параметр простого типа: `int`,
`String`, `boolean` и некоторых других.

```java
@ParameterizedTest(name = "Некорректный возраст: {0}")
@ValueSource(ints = {-1, -10, -100})
void invalidAgeIsRejected(int age) {
    ValidationResult result = validator.validateAge(age);

    assertFalse(result.isValid());
}
```

JUnit выполнит этот метод три раза:

```text
invalidAgeIsRejected(-1)
invalidAgeIsRejected(-10)
invalidAgeIsRejected(-100)
```

Вариант атрибута зависит от типа параметра:

```java
@ValueSource(ints = {1, 2, 3})
@ValueSource(strings = {"admin", "manager"})
@ValueSource(booleans = {true, false})
```

Не ставьте рядом `@Test`. Для такого метода роль тестовой аннотации уже выполняет
`@ParameterizedTest`.

### Как решить задание 4

В методе уже есть параметр `int bookingId`, вызов API и готовая проверка статуса.
Не хватает только управления запуском со стороны JUnit:

1. Добавьте над методом аннотацию параметризованного теста.
2. В её `name` запишите шаблон из условия.
3. Добавьте источник трёх `int` из условия.
4. Удалите `@Disabled`.
5. Запустите метод и убедитесь, что IntelliJ показывает три запуска.

Ни `response.statusCode()`, ни тело теста менять не требуется. Смысл задания —
научить JUnit трижды вызвать один готовый сценарий и передать ему `bookingId`.

## 8. `@CsvSource`: несколько значений на один запуск

Если на один запуск нужны логин, пароль, ожидаемый статус и ожидаемый признак
токена, одного `@ValueSource` уже недостаточно. Для небольшой таблицы простых
значений используют `@CsvSource`.

```java
@ParameterizedTest(name = "Роль {0}: доступ {2}")
@CsvSource({
        "admin,  /admin,   true",
        "viewer, /admin,   false",
        "viewer, /profile, true"
})
void accessDependsOnRole(String role, String path, boolean expectedAccess) {
    boolean actualAccess = accessService.canOpen(role, path);

    assertEquals(expectedAccess, actualAccess);
}
```

Каждая строка — один запуск. Значения сопоставляются с параметрами метода слева
направо:

```text
первая колонка  → String role
вторая колонка  → String path
третья колонка  → boolean expectedAccess
```

JUnit сам:

- читает следующую строку;
- преобразует текст `true` или `false` в `boolean`;
- передаёт значения в параметры метода;
- запускает тело;
- сохраняет результат отдельным тестом.

Именно поэтому это задача по JUnit. Без параметризации пришлось бы вручную писать
цикл или три одинаковых теста, а ошибка выглядела бы как падение одного общего
метода. С JUnit видно, на каком конкретно наборе данных произошла ошибка.

### Как решить задание 5

В `task05` уже написаны вызов авторизации, безопасная проверка наличия токена и
два итоговых `assert`. Их менять не нужно.

Нужно сделать только JUnit-часть:

1. Добавить аннотацию параметризованного теста и шаблон имени из условия.
2. Добавить `@CsvSource`.
3. Записать в нём три строки из условия.
4. Проверить порядок колонок по параметрам метода.
5. Удалить `@Disabled` и увидеть три отдельных запуска.

Для первой строки соответствие будет таким:

```text
admin       → username
password123 → password
200         → expectedStatus
true        → tokenExpected
```

Следующие строки собираются по той же схеме. Подумать здесь нужно над переводом
условия в таблицу и правильным порядком столбцов, а не над разбором ответа API.

## 9. Проверка `null`, пустой строки и пробела

Для обязательного текстового поля обычно нужны как минимум три разных проверки:

```text
null  — значения нет вообще
""    — строка есть, но в ней ноль символов
" "   — строка содержит пробел и не является пустой для Java
```

JUnit предоставляет отдельные источники:

```java
@ParameterizedTest
@NullSource
@EmptySource
@ValueSource(strings = {" "})
void invalidNameIsRejected(String name) {
    // один и тот же сценарий запускается для трёх разных значений
}
```

`@NullSource` передаёт `null`. `@EmptySource` передаёт пустую строку. Пробел нужно
указать явно в `@ValueSource`, потому что пустая строка и пробел — разные данные.

### Как решить задание 6

Тело метода полностью готово. Добавьте аннотацию параметризованного теста с именем
из условия и соедините три показанных выше источника. После удаления `@Disabled`
JUnit должен показать три запуска, каждый со статусом 400.

## 10. Понятные имена и теги

### `@DisplayName`

Имя Java-метода удобно программисту, но в отчёте иногда нужен понятный текст:

```java
@Test
@DisplayName("Сервис возвращает статус готовности")
void healthCheckReturnsOk() {
    // тест
}
```

`@DisplayName` меняет только отображение. Он не запускает тест и не влияет на его
логику.

### `@Tag`

Тег объединяет тесты в наборы:

```java
@Test
@Tag("smoke")
void serviceIsAvailable() {
    // тест
}
```

Практичные теги:

- `smoke` — быстрые критичные проверки;
- `regression` — более полный регресс;
- `api` — тесты API;
- `slow` — долгие сценарии.

Тег не делает тест быстрым или медленным. Это простая метка, по которой система
сборки может включить или исключить тест.

### Как решить задание 7

Вызов API и проверка статуса уже написаны. Добавьте над методом аннотацию для
отображаемого имени и отдельную аннотацию с тегом. Значения обеих аннотаций даны
в условии задания. После запуска посмотрите, как изменилось имя теста в IntelliJ.

## 11. Итоговый сценарий и задание 8

В рабочем API-тесте часто соединяются несколько изученных частей:

```text
@BeforeEach создаёт свежий DTO
        ↓
тест отправляет запрос и сохраняет ID
        ↓
assertAll проверяет один полный сценарий
        ↓
@AfterEach удаляет созданную запись
```

В `task08` подготовка, запросы, чтение ответа и регистрация ID для очистки уже
написаны. Нужно собрать проверки одного сценария внутри `assertAll`.

Используйте переменные, которые находятся прямо над `assertAll`:

```text
createResponse → статус создания
bookingId      → созданный идентификатор
readResponse   → статус чтения
saved          → прочитанная запись
standardBooking→ исходные ожидаемые данные
```

Для сравнения полей DTO обращайтесь к компонентам record через методы со скобками:
`saved.firstname()` и `standardBooking.firstname()`. Скобки нужны потому, что
компоненты Java `record` доступны через автоматически созданные методы.

Задание считается выполненным, когда все проверки находятся внутри одного
`assertAll`, а `task08` проходит вместе с настроенными заданиями 2 и 3.

## 12. Зачем в заготовках стоит `@Disabled`

`@Disabled` временно запрещает запуск незавершённого теста:

```java
@Test
@Disabled
void unfinishedTest() {
}
```

Рабочий порядок для этого урока:

1. Прочитайте условие и соответствующий раздел теории.
2. Допишите проверки или аннотации задания.
3. Удалите `@Disabled` только у этого задания.
4. Запустите его отдельно.
5. После решения блока запустите весь класс.

В заданиях 4–6 метод сначала не является тестом: на нём ещё нет
`@ParameterizedTest`. Поэтому зелёная кнопка возле него может появиться только
после добавления этой аннотации. Это нормальное поведение IntelliJ.

В рабочем проекте `@Disabled` не используют как вечное хранилище сломанных тестов.
У отключения должна быть понятная причина и задача на исправление.

## 13. Частые ошибки

### Параметризованный метод не запускается

Проверьте сразу три вещи:

1. Есть `@ParameterizedTest`, а не обычный `@Test`.
2. Есть хотя бы один источник данных.
3. Удалён `@Disabled`.

### JUnit сообщает, что не может передать параметр

Обычно над методом с параметром случайно поставили `@Test`. Обычный `@Test` не
знает, откуда взять `int bookingId` или `String username`. Нужны
`@ParameterizedTest` и источник данных.

### `CsvSource` передал значения не туда

Проверьте строки слева направо. Количество колонок должно совпадать с количеством
параметров метода, а их порядок — с порядком параметров.

### Вместо трёх запусков виден один

Не пишите ручной цикл внутри обычного `@Test`. Источник должен находиться над
методом рядом с `@ParameterizedTest`. Тогда каждый набор станет отдельным запуском.

### `@BeforeEach` или `@AfterEach` не выполняется

Импорт класса аннотации ничего не включает. Сама аннотация должна стоять прямо
над нужным методом.

### Тесты проходят отдельно, но мешают друг другу вместе

Чаще всего тесты используют общие изменяемые данные или не очищают созданные
записи. Каждый тест должен получать свежие данные и регистрировать созданные ID
для очистки.

## 14. Что намеренно не включено в первый блок

Здесь нет фабрик тестов, собственных расширений JUnit и `@MethodSource`. Эти
возможности полезны в подходящих задачах, но не нужны для освоения ежедневной
основы. Сначала важно уверенно использовать:

- обычный `@Test`;
- проверки и `assertAll`;
- `@BeforeEach` и `@AfterEach`;
- `@ParameterizedTest`;
- `@ValueSource` и `@CsvSource`;
- источники `null` и пустых значений;
- `@DisplayName` и `@Tag`.

## Критерий готовности

После блока вы должны уметь своими словами объяснить:

1. Почему JUnit не отправляет HTTP-запрос, но всё равно управляет API-тестом.
2. Когда подготовку выносить в `@BeforeEach`.
3. Почему очистку надёжнее выполнять в `@AfterEach`.
4. Кто и откуда передаёт значение в параметр тестового метода.
5. Когда выбирать `@ValueSource`, а когда `@CsvSource`.
6. Что означают `{0}`, `{1}` и `{index}` в имени параметризованного запуска.
7. Почему три набора параметров видны как три отдельных результата.
8. Для чего нужны `@DisplayName`, `@Tag` и `assertAll`.

Если эти ответы понятны и восемь заданий проходят вместе, практическая основа
JUnit для повседневных API-автотестов собрана.
