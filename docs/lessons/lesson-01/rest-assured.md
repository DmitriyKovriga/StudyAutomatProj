# REST Assured: от первого запроса до рабочего теста API

Материал рассчитан на человека, который уже понимает, зачем нужны тесты, но только
начинает автоматизировать REST API на Java. После чтения должно быть понятно не только
что написать, но и зачем нужна каждая часть теста.

Задания находятся в `RestAssuredTasksTest.java`. В них нет готовых запросов. Теория
показывает те же строительные детали на других примерах, поэтому решение придётся
собрать самостоятельно.

## 1. Как четыре инструмента участвуют в одном тесте

В нашем проекте обязанности разделены так:

- **JUnit** находит и запускает методы с `@Test`.
- **REST Assured** собирает и отправляет HTTP-запрос, затем принимает ответ.
- **Jackson** превращает Java-объект в JSON и JSON обратно в Java-объект.
- **AssertJ** сравнивает фактический результат с ожидаемым.

Это один последовательный процесс:

```text
JUnit запускает тест
        ↓
REST Assured отправляет HTTP-запрос
        ↓
сервер возвращает статус, заголовки и тело
        ↓
Jackson превращает JSON в Java-объект
        ↓
AssertJ проверяет статус и данные
        ↓
JUnit отмечает тест пройденным или упавшим
```

REST Assured не выполняет проверки сам по себе. Вызов `get(...)` только отправляет
запрос. Проверка появляется тогда, когда мы сравниваем полученный результат с ожидаемым.

## 2. Что нужно понимать про HTTP

У HTTP-запроса есть несколько основных частей.

```text
POST https://example.test/users/15?notify=true
     └────── адрес сервера ──────┘ └параметр запроса┘

Content-Type: application/json
Authorization: Bearer abc123

{
  "name": "Ivan"
}
```

- `POST` — HTTP-метод, то есть вид действия.
- `https://example.test` — основной адрес сервера.
- `/users/15` — путь к ресурсу.
- `15` — параметр пути: он указывает конкретного пользователя.
- `notify=true` — параметр запроса: он уточняет действие или фильтрует данные.
- `Content-Type` сообщает серверу формат тела запроса.
- `Authorization` передаёт данные авторизации.
- JSON между фигурными скобками — тело запроса.

У ответа сервера тоже есть части:

```text
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 15,
  "name": "Ivan"
}
```

- статус `200` сообщает общий результат операции;
- заголовки содержат техническую информацию;
- тело содержит данные или описание ошибки.

### Основные HTTP-методы

| Метод | Обычный смысл |
|---|---|
| `GET` | получить данные |
| `POST` | создать запись или выполнить отдельное действие |
| `PUT` | полностью заменить существующую запись |
| `PATCH` | изменить только часть полей |
| `DELETE` | удалить запись |

Статусы нельзя угадывать по привычке. Всегда сверяйтесь с документацией конкретного API.
Например, Restful Booker после удаления возвращает `201`, хотя в другом проекте это мог
бы быть `200` или `204`.

## 3. Как читать цепочку REST Assured

Рассмотрим запрос к вымышленному API пользователей:

```java
Response response = given()
        .spec(BASE_SPEC)
        .contentType(ContentType.JSON)
        .body(userRequest)
        .when()
        .post("/users");
```

Читайте его сверху вниз как предложение:

1. `given()` — начинаем описывать запрос.
2. `.spec(BASE_SPEC)` — подключаем общие настройки сервера.
3. `.contentType(ContentType.JSON)` — сообщаем, что отправляем JSON.
4. `.body(userRequest)` — кладём Java-объект в тело запроса.
5. `.when()` — закончили подготовку и переходим к действию.
6. `.post("/users")` — отправляем POST по указанному пути.
7. Результат сохраняем в `Response`.

Переносы строк не меняют работу кода. Они нужны, чтобы запрос было удобно читать.

### Что хранит `BASE_SPEC`

В заданиях уже подготовлена общая спецификация:

```java
private static final RequestSpecification BASE_SPEC = new RequestSpecBuilder()
        .setBaseUri(BASE_URL)
        .setAccept(ContentType.JSON)
        .build();
```

Она хранит настройки, одинаковые для многих запросов:

- адрес сервера;
- ожидаемый формат ответа.

Спецификация ничего не отправляет. Она похожа на заранее заполненную часть формы.
Конкретный тест всё равно должен выбрать путь, HTTP-метод, параметры, авторизацию и тело.

## 4. Как работать с `Response`

`Response` — полный ответ сервера. Из него можно получить разные части:

```java
int status = response.statusCode();
String contentType = response.contentType();
String bodyAsText = response.asString();
```

Если ответ содержит JSON одного объекта, его обычно преобразуют в DTO:

```java
UserResponse user = response.as(UserResponse.class);
```

Проверки можно разделить на два слоя:

```java
assertThat(response.statusCode()).isEqualTo(200);
assertThat(user.name()).isEqualTo("Ivan");
```

Первая строка проверяет работу HTTP-операции. Вторая проверяет данные предметной области.
Для хорошей диагностики обычно нужны обе.

## 5. DTO и роль Jackson

DTO — простой Java-объект для передачи данных. В учебном проекте удобно использовать
`record`, потому что он сразу создаёт конструктор и методы чтения полей.

```java
record UserRequest(String name, boolean active) {
}

record UserResponse(int id, String name, boolean active) {
}
```

### Превращение Java в JSON

Когда в REST Assured написано:

```java
.contentType(ContentType.JSON)
.body(userRequest)
```

Jackson видит Java-объект и создаёт из него JSON. Этот процесс называется
**сериализацией**.

### Превращение JSON в Java

Когда написано:

```java
UserResponse user = response.as(UserResponse.class);
```

Jackson читает JSON и создаёт Java-объект. Это называется **десериализацией**.

Запоминать термины проще так:

```text
Java → JSON: сериализация, отправляем
JSON → Java: десериализация, читаем
```

Имена компонентов `record` должны совпадать с именами полей JSON. Если сервер возвращает
`first_name`, а в Java записано `firstName`, потребуется `@JsonProperty("first_name")`.
В Restful Booker имена совпадают, поэтому дополнительные аннотации не нужны.

## 6. Один объект, список и объект-обёртка

### Один JSON-объект

```json
{
  "id": 10,
  "name": "Ivan"
}
```

Для него достаточно класса:

```java
UserResponse user = response.as(UserResponse.class);
```

### JSON-массив

```json
[
  {"id": 10},
  {"id": 11}
]
```

Для списка DTO используется `TypeRef`, чтобы Jackson не потерял тип элементов:

```java
List<UserIdResponse> users = response.as(
        new TypeRef<List<UserIdResponse>>() {
        }
);
```

### Объект-обёртка

Иногда сервер возвращает ID и созданный объект вместе:

```json
{
  "id": 10,
  "user": {
    "name": "Ivan",
    "active": true
  }
}
```

Тогда Java-модель повторяет структуру JSON:

```java
record CreateUserResponse(int id, UserResponse user) {
}
```

Сначала смотрите на форму JSON, затем проектируйте DTO. Не пытайтесь поместить весь
ответ в один плоский класс, если в JSON есть вложенные объекты.

## 7. Параметр пути и параметр запроса

### Параметр пути

Он выбирает конкретную запись:

```java
Response response = given()
        .spec(BASE_SPEC)
        .pathParam("id", userId)
        .when()
        .get("/users/{id}");
```

REST Assured заменит `{id}` фактическим значением `userId`.

### Параметр запроса

Он фильтрует список или добавляет настройку операции:

```java
Response response = given()
        .spec(BASE_SPEC)
        .queryParam("name", expectedName)
        .when()
        .get("/users");
```

В результате адрес будет похож на `/users?name=Ivan`.

Короткое правило:

```text
/users/{id}     — найти конкретную запись
/users?name=... — отфильтровать список
```

## 8. Тело запроса и `Content-Type`

Для `GET` тело обычно не требуется. Для `POST`, `PUT` и `PATCH` тело используется часто.

```java
given()
        .spec(BASE_SPEC)
        .contentType(ContentType.JSON)
        .body(requestDto)
        .when()
        .post("/users");
```

`Accept` и `Content-Type` отвечают на разные вопросы:

- `Accept` — какой формат ответа мы хотим получить;
- `Content-Type` — какой формат данных мы отправляем.

Если забыть `Content-Type: application/json`, сервер может не распознать тело или вернуть
ошибку, даже если сам JSON выглядит правильно.

## 9. Авторизация с токеном

В Restful Booker работа с защищёнными операциями состоит из двух шагов.

### Шаг 1. Получить токен

В `/auth` отправляются логин и пароль. В ответ приходит JSON:

```json
{
  "token": "abc123"
}
```

Ему соответствует DTO:

```java
record TokenResponse(String token) {
}
```

### Шаг 2. Передать токен в защищённый запрос

Restful Booker ожидает токен в cookie:

```java
given()
        .spec(BASE_SPEC)
        .cookie("token", token)
        .pathParam("id", userId)
        .contentType(ContentType.JSON)
        .body(updatedUser)
        .when()
        .put("/users/{id}");
```

Важно различать:

- **аутентификация** отвечает на вопрос «кто вы?» и выдаёт токен;
- **авторизация** отвечает на вопрос «можно ли вам выполнить это действие?».

Получить токен недостаточно. Его ещё нужно передать в запрос, которому нужна защита.

## 10. Практический смысл POST, GET, PUT, PATCH и DELETE

### POST: создать запись

Обычная проверка:

1. подготовить DTO запроса;
2. отправить POST;
3. проверить статус;
4. получить ID;
5. сравнить возвращённые данные с отправленными;
6. сохранить ID для последующего удаления.

### GET: прочитать созданную запись

GET после POST важен потому, что ответ POST ещё не доказывает сохранение в базе. Сервер
может вернуть правильные данные, но не записать их. Контрольный GET обнаружит такую ошибку.

```text
DTO запроса → POST → ID → GET по ID → сохранённый DTO → сравнение
```

### PUT: полностью заменить запись

В PUT обычно отправляется полный объект. После PUT полезно сделать отдельный GET и
сравнить фактически сохранённые данные с ожидаемым объектом.

### PATCH: изменить отдельные поля

Для маленького частичного изменения удобно использовать `Map`:

```java
Map<String, Object> changes = Map.of("active", false);
```

После PATCH проверяйте две вещи:

- нужное поле изменилось;
- остальные важные поля не испортились.

### DELETE: удалить запись

Успешный ответ DELETE ещё не доказывает отсутствие записи. Поэтому после удаления
выполняют GET по тому же ID и ожидают статус «не найдено».

## 11. Что именно следует проверять

Не существует правила «проверить весь JSON целиком в каждом тесте». Проверки выбирают
по смыслу сценария.

### Технический результат

- ожидаемый HTTP-статус;
- формат ответа, если он важен;
- наличие обязательного тела.

### Данные

- созданный ID положительный;
- отправленные значения вернулись без искажения;
- обязательное поле не пустое;
- фильтр вернул созданную тестом запись.

### Состояние системы

- созданная запись читается отдельным GET;
- PUT действительно сохранил новые значения;
- PATCH не повредил остальные поля;
- запрещённое изменение не сохранилось;
- удалённая запись больше не находится.

Поэтому проверка только статуса часто слишком слабая. Статус `200` означает, что сервер
принял запрос, но не гарантирует правильность всех данных.

## 12. Тестовые данные и очистка

Публичным API одновременно пользуются разные люди. Нельзя брать случайную существующую
запись и предполагать, что у неё определённое имя или `depositpaid == true`.

Устойчивый тест работает иначе:

1. создаёт собственную запись;
2. получает её ID;
3. выполняет проверки только над ней;
4. удаляет её после теста.

Для уникальности в заданиях используется `UUID`:

```java
String uniqueName = "Student-" + UUID.randomUUID();
```

В `RestAssuredTasksTest` созданные ID кладутся в `createdBookingIds`. Метод с
`@AfterEach` пытается удалить их после каждого теста. Благодаря этому очистка запускается
даже тогда, когда одна из проверок упала.

Если тест сам успешно удалил запись, её ID нужно убрать из списка очистки. Иначе после
теста будет выполнена лишняя попытка удаления.

## 13. Как пользоваться готовыми вспомогательными методами

Внизу файла заданий есть несколько методов:

- `createBooking(...)` отправляет POST и возвращает полный ответ;
- `createTrackedBooking(...)` создаёт запись и запоминает её ID для очистки;
- `getBooking(...)` выполняет GET по ID;
- `getToken()` получает рабочий токен;
- `deleteBookingIfPresent(...)` используется для безопасной очистки;
- `uniqueBooking(...)` создаёт уникальные тестовые данные.

Это не скрытая магия. Откройте метод и прочитайте его сверху вниз. В реальном проекте
повторяющиеся запросы тоже постепенно выносят в клиент API, но сначала важно научиться
собирать их самостоятельно.

## 14. Маршрут по заданиям

### Задание 1. Получение токена

Нужно соединить три отдельные детали:

```java
.contentType(ContentType.JSON)
.body(credentials)
.post("путь из документации")
```

После запроса полный ответ сохраняется в `Response`, а JSON отдельно преобразуется через
`.as(...)`. Подумайте, какой DTO соответствует JSON с полем `token`.

### Задание 2. Получение списка ID

Здесь нет тела запроса. Нужны спецификация, GET и преобразование JSON-массива через
`TypeRef`. Обратите внимание: тип результата — `List<BookingIdResponse>`, а не один DTO.

### Задание 3. Создание записи

Нужны `Content-Type`, тело `expected` и POST. Ответ имеет обёртку: в нём есть ID и
вложенный объект booking. После преобразования ответа не забудьте зарегистрировать ID
для очистки.

### Задание 4. Чтение по ID

Запись уже создана вспомогательным методом. Вам нужно собрать GET с `pathParam`, затем
преобразовать ответ в DTO одной записи.

### Задание 5. Фильтрация

Используйте два `queryParam`: один для имени, второй для фамилии. Результатом снова будет
JSON-массив, поэтому понадобится `TypeRef`.

### Задание 6. Полное обновление

В одном тесте выполняются два разных запроса:

1. PUT с параметром пути, cookie, `Content-Type` и полным телом;
2. GET по тому же ID.

Тело ответа PUT и результат GET преобразуются в `BookingResponse` независимо друг от друга.

### Задание 7. Частичное обновление

Строение PATCH почти такое же, как PUT. Отличается тело: отправляется `changes`, где есть
только одно поле. После контрольного GET сравните изменённое поле и несколько сохранённых.

### Задание 8. Проверка прав

Намеренно не добавляйте `.cookie("token", ...)`. После отказа сервера выполните обычный
GET и докажите, что запрещённые данные не сохранились.

### Задание 9. Удаление

DELETE требует параметр пути и cookie. После успешного удаления удалите ID из
`createdBookingIds`, затем выполните контрольный GET.

### Задание 10. Полный жизненный цикл записи

Это сборка знакомых частей, а не новая технология:

```text
создать → прочитать → полностью обновить → прочитать → удалить → проверить отсутствие
```

Не пишите весь сценарий сразу. Заполняйте по одному шагу и после каждого шага проверяйте,
какое значение уже доступно следующему запросу.

## 15. Порядок размышления над новым тестом

Когда перед вами новый метод API, ответьте на вопросы по порядку:

1. Какой HTTP-метод и путь указаны в документации?
2. Нужны ли параметр пути или параметры запроса?
3. Нужна ли авторизация и где передаётся токен?
4. Есть ли тело запроса и какой у него JSON?
5. Как выглядит успешный JSON-ответ?
6. Нужен один DTO, список DTO или объект-обёртка?
7. Какой статус ожидается?
8. Какие данные доказывают правильность результата?
9. Нужно ли отдельным GET проверить сохранённое состояние?
10. Какие данные тест должен удалить после себя?

Если на один из вопросов нет ответа, вернитесь к документации API до написания кода.

## 16. Частые ошибки

### `response` остался равен `null`

Значит, запрос ещё не был присвоен переменной. Цепочка `given()...get()` должна завершиться
значением, которое сохраняется в `Response`.

### DTO равен `null`

Получить `Response` недостаточно. JSON нужно отдельно преобразовать:

```java
SomeResponse actual = response.as(SomeResponse.class);
```

### Сервер не понимает тело

Проверьте наличие `.contentType(ContentType.JSON)` перед `.body(...)`.

### Получен `403`

Для защищённого PUT, PATCH или DELETE проверьте, что токен получен и передан именно так,
как требует Restful Booker.

### Jackson сообщает о неизвестном поле или не может создать объект

Сравните форму JSON и DTO: имена, типы, вложенность. JSON-массив нельзя преобразовать в
один объект, а вложенный объект нельзя бездумно заменить строкой.

### Тест иногда падает на чужих данных

Не выбирайте случайный ID из общего списка для проверки конкретных значений. Создайте
собственную запись и работайте с её ID.

## Критерий готовности

Блок пройден, если вы можете без копирования готового теста:

1. прочитать описание метода API;
2. собрать запрос с путём, параметрами, телом и авторизацией;
3. сохранить полный `Response`;
4. преобразовать JSON в один DTO или список DTO;
5. проверить статус и важные данные через AssertJ;
6. после изменения выполнить контрольный GET;
7. создать собственные данные и гарантированно удалить их.

Печатный конспект остаётся дополнительной памяткой:
`docs/print/rest-assured-cheatsheet/rest-assured-cheatsheet.pdf`.
